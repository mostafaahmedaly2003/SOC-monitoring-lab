📄 Abstract

A fully integrated Security Operations Center (SOC) infrastructure built as part of the DF × C23 SOC Diploma Program. The project demonstrates a complete security monitoring and automated incident response pipeline using five core open-source tools deployed in a VMware lab environment.

The architecture integrates a Kali Linux attacker/gateway, a Windows 10 victim endpoint, Snort inline IPS/IDS, Wazuh SIEM, Shuffle SOAR, and T-Pot honeypot — all connected across a multi-adapter VMware topology.

Architecture
Traffic flow, security design principles, and SOAR automation pipeline
🔄 Traffic Flow — 7 Stages
1
Attack Initiation
Kali attacker sends traffic via eth2 Custom VMnet toward victim and honeypot segments
2
Snort Inspection
All eth2 traffic intercepted. IPS (NFQUEUE) drops malicious; IDS logs JSON alerts
3
Clean Forwarding
Inspected traffic forwarded via eth1 Bridged to Windows — only clean packets reach endpoint
4
Endpoint Monitoring
Wazuh agent collects Event Logs, process creation, FIM changes → Manager via TCP 1514
5
SIEM Correlation
Wazuh correlates Snort alerts + Windows telemetry + T-Pot data, generates unified alerts
6
SOAR Automation
Level 7+ alerts trigger Shuffle webhook → VT lookup → IP block + SendGrid email + log
7
Honeypot Trapping
T-Pot captures attacker interactions on eth2; Wazuh agent streams to central SIEM
🔒 Security Design Principles
NETWORK SEGMENTATION
Three adapters isolate attack traffic (Custom/eth2), victim channel (Bridged/eth1), and internet (NAT/eth0) — preventing lateral movement to SIEM management
DEFENSE-IN-DEPTH
Dual IPS/IDS: IPS blocks known threats before victim; IDS captures all suspicious activity for forensics analysis
HOST-LEVEL VISIBILITY
Wazuh agents detect local threats beyond network-level detection: process execution, file changes, privilege escalation, PowerShell
AUTOMATED RESPONSE
Shuffle reduces MTTR from minutes to seconds. Level 12+ alerts trigger automatic IP blocking without analyst intervention
HONEYPOT DECEPTION
T-Pot traffic is inherently suspicious — no legitimate users — providing high-confidence IOCs and attacker TTP data
CREDENTIAL MANAGEMENT
API keys stored as Docker env vars. Shuffle uses encrypted credential storage. Agent keys distributed via DF × C23 secure onboarding

⚠️ Known Limitations

Wazuh, Shuffle, and Snort on same VM creates resource contention in sustained attack scenarios
LAB SCALE ONLY
One victim endpoint vs production hundreds/thousands. Alerting volume not representative
FREE API RATE LIMITS
VirusTotal 4 req/min free tier — insufficient for production threat volumes
SELF-SIGNED TLS CERTIFICATES
Production deployment requires CA-signed certificates for all services
NO HIGH AVAILABILITY
Single point of failure throughout — clustering and redundancy needed for production
LIMITED ATTACK COVERAGE
Common attacks only; no APTs or zero-days tested. Manual rule management, no CI/CD feeds

Challenges & Solutions
Technical obstacles encountered during implementation and how they were resolved
Challenge
Snort Adapter Assignment
Unclear whether to use Bridged or Custom VMnet for IPS inline placement
Solution Applied
Custom (eth2) is correct — sits between attacker and victim, enabling genuine inline IPS blocking before traffic reaches the endpoint
Challenge
Dual IPS/IDS on One Interface
Complex NFQUEUE configuration for simultaneous inline prevention and passive detection
Solution Applied
NFQUEUE iptables rules for IPS + passive alert mode simultaneously on eth2. Both modes functional on same adapter
Challenge
Wazuh Docker RAM Usage
High memory consumption from Wazuh + Shuffle on single VM host
Solution Applied
8 GB RAM allocation to Kali VM; tuned OpenSearch JVM heap size to reduce pressure
Challenge
Webhook SSL/TLS Failures
Wazuh → Shuffle webhook calls failing due to self-signed certificate rejection
Solution Applied
Configured correct HTTPS port 3443 in Shuffle; disabled TLS verification for lab environment (not for production)
Challenge
T-Pot vs Central Wazuh Conflict
T-Pot's built-in Elastic Stack conflicted with attempting to run central Wazuh on the same host
Solution Applied
Deployed separate Wazuh agent on T-Pot VM; kept T-Pot's native Elastic Stack locally, forwarded agent data to central Wazuh
Challenge
Windows Agent Enrollment
Cross-segment connectivity issues preventing agent registration across Bridged/Custom adapters
Solution Applied
Verified IP routing between segments; added firewall exceptions for TCP 1514/1515 on both Kali and Windows
Challenge
VirusTotal Rate Limiting
Free tier restricted to 4 requests/min — insufficient for high-alert scenarios in testing
Solution Applied
Added Shuffle delay node between alerts; implemented lookup result caching to reduce API call frequency

Conclusion
Summary of achievements, validation, and program impact

This project successfully designed, implemented, and validated a fully integrated SOC infrastructure using open-source tools within the Digital Fortress × Confianza23 SOC Diploma Program.

The multi-adapter VMware topology provides realistic traffic segmentation replicating production network zones. Snort's dual IDS/IPS mode delivers both inline prevention and passive detection. Wazuh's Docker-based SIEM centralizes logs from Snort alerts, Windows agents, and T-Pot honeypot. Shuffle SOAR automates incident response via webhook-driven workflows.

While lab-scale with known limitations, the project demonstrates that open-source tools can build an effective SOC pipeline covering the Detect, Respond, and Recover functions of the NIST Cybersecurity Framework.
